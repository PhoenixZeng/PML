# PML 格式规范

[English Spec](./PML-format-spec.md) | [English README](./README.md)


## 1. PML 是什么
PML 是一种为 LLM、提示词工程和结构化文本拼接而设计的轻量块格式。

作为正式名称，PML 可以理解为 **Prompt Meta Language**。

同时，它也保留了一点有意为之的开放语义，例如：
- **Peer as a Markup Language**
- **Patchwork Mosaic Language**
- **Prompt Modular Language**
- **PML forMat Language**

这种模糊不是缺陷，而是设计的一部分。

它适合处理这样一类问题：

    你有很多段不同格式的文本
    例如 system prompt、user input、配置片段、Markdown 说明、JSON 数据、YAML 参数等
    想把它们稳定地放在同一份文档里
    同时又不希望它们在拼接、传递或二次处理时互相干扰
    更重要的是，你希望人类和 LLM AI 都可以轻易识别这种混用

PML 的目标不是取代 JSON、YAML 或 Markdown。
相反，它更像这些格式外面的一层“信封”：

1. 用清晰的块边界把长文本拆开。
2. 给每段正文附一个可选类型，说明它将来应如何理解。
3. 在解析时保留正文原文，不提前进入正文内部语义。
4. 让一份文档既适合人类读，也适合程序和 LLM AI 稳定处理。

在现代提示词工程中通常采用Markdown加代码块来解决这个问题

但是它有一些缺陷：
- Markdown难以自嵌套
- 多个Markdown拼接时层级混乱

而此格式正是为了解决这个问题

另一个作用是如果你使用 PML 来编写README或配置文件，它甚至可以直接充当具有大量注释说明的配置文件

> PML 不只是一个“数据格式”，更是一种用来组织可提示、可分层、可混合文本内容的格式(包括混合 PML 自身)。

它不负责表达对象树，也不负责解析正文内部语义。
它只做三件事：

1. 把长文本切成一组有名字的块。
2. 给每个块附一个可选类型。
3. 稳定保留正文原文。

因此，PML 特别适合这些场景：

1. 提示词模板与多段 prompt 组织
2. agent 中间态载荷
3. 多种文本格式的拼接与封装
4. 需要“先切块、后处理”的结构化文本工作流
5. 具有大量注释说明的配置文件或多种格式不同的配置文件共存

## 2. 最小例子

```text
[SYSTEM]
你是一个严谨助手。

[CONFIG:yaml]
lang: zh-CN
tone: concise
```

按 PML 解析后，可以先直觉地理解成下面这样的路径树模型简化示意：

```json
{
  "SYSTEM": {
    "type": "text",
    "content": "你是一个严谨助手。"
  },
  "CONFIG": {
    "type": "yaml",
    "content": "lang: zh-CN\ntone: concise"
  }
}
```

这里最重要的一点是：
`content` 始终先按原始字符串返回，PML 不会继续把它解析成 YAML 或别的结构。
如果你需要，你可以另行解释。这也意味着你可以胡乱的往 PML 放置各种格式。
更完整、可保真或可逆的数据表示方式，见第 16 节。

## 3. 核心原则

PML 的核心原则只有四条：

1. 文档由有序块序列组成，不是树。
2. 解析器一次只解析当前层。
3. 每个块的正文都先按原始字符串处理。
4. 块名可以有分段和附加标记，但 PML 核心不规定它们的结构投影语义。

这意味着：

```text
[A.B]
[A.1]
[1.1.1]
```

在 PML 核心里都只是不同名字的块。
在解析上 PML 不会自动把它们变成对象、数组或章节树。
不过如果你真的需要，PML 也可以满足你，见第 16 节。

## 4. 语法

### 4.1 开启块头

开启块头有两种基本形式：

```text
[NAME]
[NAME:TYPE]
```

### 4.2 关闭块

关闭块也是两种基本形式：

```text
[/NAME]
[/NAME:TYPE]
```

关闭块是可选的。
只有当作者需要显式边界时才需要写关闭块。

### 4.3 名称

`NAME` 是名称部分。命名风格上，`NAME` 建议优先使用大写或清晰可读的标题式命名；
它本身可以带分段，也可以在末尾带一个 `#...` 后缀。

推荐词法如下：

下面这组正则是简要实现，适合快速上手或仅处理常见 ASCII/CJK 场景。
如果需要更完整的跨语言 Unicode 支持，应再参考后面的“完整语义”规则。

主名：

```text
[A-Za-z0-9_\u4e00-\u9fff]+(\.[A-Za-z0-9_\u4e00-\u9fff]+)*
```

完整名称部分：

```text
[A-Za-z0-9_\u4e00-\u9fff]+(\.[A-Za-z0-9_\u4e00-\u9fff]+)*(#[A-Za-z0-9_\u4e00-\u9fff]+)?
```

例如：

```text
A
SYSTEM
PROMPT.INPUT
a.b.c
章节.一
系统提示
A.1
1.1.1
A#raw
系统提示#外层
PROMPT.INPUT#outer
```

后面的 `#...` 现在只需要先把它看成名称的一部分。
它为什么存在，后面再单独解释。

这里使用 CJK 范围 `[\u4e00-\u9fff]` 表示中日韩文字符。
如果在具体编程语言的字符串字面量中书写这个正则表达式，反斜杠可能需要写成 `\\u`，具体取决于该语言的字符串转义规则。

如果需要更完整的 Unicode 版本，可以把名称理解成：

```text
NAME = SEGMENT ("." SEGMENT)* ("#" SEGMENT)?
```

其中 `SEGMENT` 的规则如下：

1. 首字符允许：
   - Unicode 字母
   - Unicode 数字
   - `_`
2. 后续字符允许：
   - Unicode 字母
   - Unicode 数字
   - Unicode 组合标记
   - `_`
3. `#` 后面的部分与普通段使用完全相同的规则，不额外放宽。

准确来说在Unicode类别中：
1. 起始字符：L*、N*、_
2. 后续字符：L*、N*、M*、_
3. 拒绝：P*、Z*、S*、C*


因此，一个完整实现应当拒绝这些字符类别：

1. 标点符号
2. 空格与其他空白分隔符
3. 控制字符
4. 符号字符，例如 emoji、商标符号、数学符号等
5. 私有区、代理项和未分配字符

如果某种编程语言方便直接访问 Unicode 字符类别，推荐按上面的类别规则实现；
如果不方便，也可以先采用上面的简要正则表达式，作为一个实用但不完整的近似实现。

### 4.4 类型

`TYPE` 是可选的正文格式标签。

规则如下：

1. 如果省略 `TYPE`，默认类型是 `text`。
2. 类型比较大小写不敏感。
3. 解析前应先做类型规范化，推荐统一成小写。
4. 核心规范只定义一个别名特例：`md = markdown`。
5. 至于具体是什么格式，答案是任何格式都可以，包括你自创的格式。
6. `TYPE` 只允许使用 ASCII 字母、数字、`_`、`-`。

命名风格上`TYPE` 建议优先使用小写。
常见写法例如：`text`、`markdown`、`json`、`yaml`、`toml`、`ini`、`prompt-template`。

例如：

```text
[NOTE]
hello
[FOO:abc]
bar
```

等价于：

```text
[NOTE:text]
hello
[FOO:abc]
bar
```

而：

```text
[DOC:MD]
hello
[/DOC:markdown]
```

也是合法的。

PML 不定义其他内建类型别名表。

## 5. 两种块形态

PML 只有两种块形态：

1. 短块
2. 配对块

### 5.1 短块

短块只有开启块头，没有关闭块。

它的正文从块头下一行开始，一直延伸到：

1. 下一个合法开启块头
2. 或文件结束

终止边界前的分隔换行不属于正文。

示例：

```text
[TITLE]
Weekly Report

[META:json]
{"version":1}
```

### 5.2 配对块

配对块既有开启块头，也有匹配的关闭块。

正文从块头下一行开始，终止于匹配关闭块。

关闭块前的分隔换行不属于正文。

示例：

```text
[PAYLOAD:json]
{
  "name": "demo",
  "enabled": true
}
[/PAYLOAD]
```

## 6. 解析规则

### 6.1 控制行必须独占一行

PML 只在整行匹配控制语法时，才把该行识别为控制行。

例如：

```text
[A]
[/A]
[A2:type]
[/A2:type]
[A3#tag:type]
[/A3#tag:type]
```

推荐实现时先去掉行尾 `\r`，再按整行匹配。

### 6.2 正文边界与末尾换行

正文区由一组正文行组成。

正文从开启块头之后的第一行开始，终止于：

1. 匹配的关闭块
2. 下一个合法开启块头
3. 或 EOF

终止边界前的分隔换行不属于正文。
换句话说，控制行本身以及它前面的行分隔符是块边界的一部分，不是正文内容。
EOF 前的最后一个分隔换行也按同样规则处理。

因此：

```text
[A]
hello
[/A]
```

`A` 的正文是：

```text
hello
```

而：

```text
[A]
hello

[/A]
```

`A` 的正文是：

```text
hello

```

也就是字符串 `hello\n`。

空正文示例：

```text
[A]
[/A]
```

`A` 的正文是空字符串。

一个空行正文示例：

```text
[A]

[/A]
```

`A` 的正文是字符串 `\n`。

短块到 EOF 也完全一样。

```text
[A]
hello
```

如果文件在 `hello` 后立即结束，则 `A` 的正文是 `hello`。

```text
[A]
hello

```

如果文件在空白行后结束，则 `A` 的正文是 `hello\n`。

这条规则适用于所有正文类型，包括 `text`、`markdown`、`json`、`yaml`、`toml` 和 `ini`。
PML 仍然保留正文区内的原始内容，只是不把终止边界前的分隔换行算作正文。

### 6.3 配对块优先

如果某个开启块后面能找到匹配关闭块，那么它按配对块解析，并吞掉中间全部内容。

例如：

```text
[A]
line 1
[B]
line 2
[/A]
```

当前层只得到一个 `A` 块，它的正文是：

```text
line 1
[B]
line 2
```

### 6.4 找不到关闭块时按短块解析

如果某个开启块后面找不到匹配关闭块，那么它按短块处理，正文终止于下一个合法开启块头或 EOF。

### 6.5 关闭块匹配规则

关闭块匹配只需要记住三条：

1. `NAME` 必须完全匹配。
2. 如果关闭块写了 `TYPE`，那么它必须和开启块的规范化类型一致。
3. 如果关闭块没写 `TYPE`，则这次关闭不校验类型。

例如：

```text
[A]
...
[/A:text]
```

合法，因为 `[A]` 的默认类型是 `text`。

```text
[A:yaml]
...
[/A]
```

合法，因为 `[/A]` 不校验类型。

```text
[A:yaml]
...
[/A:json]
```

不合法。

### 6.6 名称中的 `#...` 也参与匹配

如果开启块和关闭块的名称部分不同，就不能互相匹配。

因此：

```text
[A#outer]
...
[/A]
```

不合法。

而：

```text
[A#outer]
...
[/A#outer]
```

合法。

## 7. 为什么名称里允许 `#...`

先看一个场景。

作者想包一段 PML 示例，而这段示例内部刚好也用了同名块：

```text
[A]
[A]
inner
[/A]
[/A]
```

这会产生边界冲突：
解析器无法知道第一个 `[/A]` 到底是正文内容，还是外层块的结束位置。

当然，作者可以直接改外层名字：

```text
[A_outer]
[A]
inner
[/A]
[/A_outer]
```

但这样会为了避开边界冲突，改掉原本真正想保留的名字。

因此 PML 允许在名称后面加一个 `#...`：

```text
[A#outer]
[A]
inner
[/A]
[/A#outer]
```

这有两个好处：

1. 主名仍然是 `A`。
2. 匹配时 `A` 和 `A#outer` 又天然不是同一个名称。

所以 `#...` 的作用不是增加结构语义，而是把边界变得更具体，同时尽量保留原始主名。

## 8. PML 不提供转义语法

PML 不定义任何“让控制行失效”的转义字符。

原因很简单：
PML 本来就是为了粘合多种语法，如果再加一层自己的转义规则，反而更容易制造新的冲突。

因此 PML 的选择是：

1. 不做正文转义
2. 不做正文逃逸
3. 冲突通过调整边界解决，例如改名称或加 `#...`

## 9. 自嵌套表达

PML 可以承载另一份 PML 文档，但当前层不会递归解析它。

例如：

```text
[OUTER#v1:PML]
[INNER]
hello
[/INNER]
[/OUTER#v1]
```

在最外层，这整段内部文本都会作为 `OUTER#v1` 的正文返回。
如果调用方愿意，再把这段正文作为 PML 做下一轮解析。

这就是 PML 所说的“表达上允许递归承载，解析上不要求递归下降”。

## 10. 严格性与错误

PML 只有严格模式，没有宽松模式。

以下情况都应直接报错：

1. 非法开启块头，例如 `[A key="value"]`
2. 非法关闭块，例如 `[/A extra]`
3. 游离关闭块，例如单独出现的 `[/A]`
4. 类型不匹配，例如 `[A:yaml] ... [/A:json]`
5. 名称不匹配，例如 `[A#x] ... [/A#y]`

## 11. 推荐写法

1. 普通正文优先用短块。
2. 当正文里可能出现 PML 风格行时，优先用配对块。
3. 当边界可能冲突时，优先在名称后加 `#...`，而不是污染主名。
4. 数据放到正文里，正文格式可以是 `markdown`、`yaml`、`json` 或其他自定义类型。

## 12. 参考解析流程

可以按下面的顺序实现解析：

1. 逐行扫描文档。
2. 识别合法开启块头。
3. 对开启块类型做规范化：
   省略时补成 `text`，比较时大小写不敏感，`md` 规范化为 `markdown`。
4. 确定正文终止边界：匹配关闭块、下一个合法开启块头或 EOF。
5. 取开启块后的正文行，并排除终止边界前的分隔换行。
6. 如果需要配对块校验，则向后找名称匹配、且类型校验通过的关闭块；找不到则生成短块。

## 13. 非目标

PML 不试图提供这些能力：

1. 任意属性系统
2. 自动对象投影
3. 自动数组投影
4. 单次解析直接构造整棵树
5. 自动解析 `yaml`、`json`、`markdown` 或此格式嵌套的 `PML` 正文
6. 取代 Markdown、JSON 或 YAML

## 14. 总结

PML 的定义可以收成九条：

1. 文档由有序块序列组成。
2. 开启块头写作 `[NAME]` 或 `[NAME:TYPE]`。
3. 关闭块写作 `[/NAME]` 或 `[/NAME:TYPE]`。
4. `NAME` 本身可以带分段，也可以带 `#...` 后缀。
5. 解析器一次只解析一层。
6. 每个块的正文都先按原始字符串返回。
7. 终止边界前的分隔换行不属于正文，末尾空行才表示正文末尾换行。
8. 关闭块里的 `TYPE` 是可选校验项。
9. PML 是严格格式，不提供正文转义机制。

PML 适合提示词模板、结构化文本封装、中间态载荷和片段拼接场景。
它关心的重点不是富数据建模，而是“块边界稳定、正文保真、后续处理简单”。

## 15. 给 LLM 的快速提示词

下面给出两版可直接复用的提示词。

### 15.1 简单版

适合只处理常规块切分，不涉及嵌套、`#...` 或其他特殊边界场景。

```text
[说明]
你需要理解一种叫 PML 的轻量块式文本格式。

[规则]
1. 文档由一组按顺序出现的块组成。
2. 开启块头写作 [NAME] 或 [NAME:TYPE]，如果省略 TYPE 表示默认类型是 text。
3. 可选的关闭块写作 [/NAME] 或 [/NAME:TYPE]。
4. 如果一个块后面存在匹配关闭块，则它是配对块；如果不存在，则它是短块。
5. 关闭块如果省略 TYPE，则表示不校验类型；如果写了 TYPE，则要与开启块类型一致。
6. 类型可以是其他任意类型如json、yaml等，其正文始终按其格式理解，解析正文内部语义。

[总结]
把 PML 理解成一个有序块列表。
每个块包含：
- name
- type
- content
此文即采用 PML 格式编写。
```

### 15.2 完整版

适合需要处理嵌套承载、`#...` 边界限定和完整匹配规则的场景。

```text
[说明]
你需要理解一种叫 PML 的轻量块式文本格式。
PML 的输出是有序块列表，不是树。
PML 内可以嵌套承载另一份 PML 文档，例如外层块的正文里再放 PML。你应该分层理解。

[语法规则]
1. 开启块头写作 [NAME] 或 [NAME:TYPE]，如果省略 TYPE 表示默认类型是 text。
2. 可选的关闭块写作 [/NAME] 或 [/NAME:TYPE]。
3. NAME 本身可以带分段，也可以带 #... 后缀，例如 A.B、A.1、A#outer。
4. 类型比较大小写不敏感，md 视为 markdown。类型可以是其他任意类型如json、yaml等
5. 正文始终按其格式理解，解析正文内部语义。

[解析规则]
1. 控制行必须独占一行。
2. 如果某个开启块后面能找到匹配关闭块，则按配对块解析，并吞掉中间全部内容。
3. 如果找不到匹配关闭块，则按短块解析，到下一个合法开启块头或 EOF 为止。
4. 关闭块匹配时，NAME 必须完全一致。不同的分段或#后缀视为不同的名称。
5. 如果关闭块写了 TYPE，则它必须与开启块的规范化类型一致。
6. 如果关闭块没写 TYPE，则这次关闭不校验类型。

[关于#标记]
1. 你可以先把 #... 视为名称的一部分。它不是转义语法，也不是结构字段。
2. 它的作用是让边界更具体，避免同名块在嵌套承载时发生关闭冲突。

[总结]
把 PML 理解成一个有序块列表。
每个块包含：
- name
- type
- content
此文即采用 PML 格式编写。
```

## 16. 可选的数据化解释

虽然 PML 不试图取代 JSON、YAML 等数据格式，但它仍然可以被映射成程序更容易处理的数据结构。

这种能力不是 PML 核心解析规则的一部分，而是建立在“块名 + 正文”之上的一种可选解释层。
之所以把它放在最后讨论，是为了避免它反过来污染 PML 的基础定位：
PML 首先仍然是块式文本封装格式，其核心任务是稳定切块、保留正文、延后解释。

这里推荐两种数据化模型：

1. 块序模型
2. 路径树模型

### 16.1 块序模型

块序模型与 PML 核心解析最一致。

它的特点是：

1. 保留原始块顺序
2. 不尝试把块名自动投影成对象树
3. 最接近解析器运行时直接得到的结构
4. 最容易做到无损或近似无损转换

推荐结构如下：

```json
{
  "blocks": [
    {
      "name": "SYSTEM",
      "type": "text",
      "content": "你是一个严谨助手。"
    },
    {
      "name": "CONFIG",
      "type": "yaml",
      "content": "lang: zh-CN\ntone: concise"
    }
  ]
}
```

如果只需要一个稳定、简单、可保真的数据表示，块序模型通常已经足够。

### 16.2 路径树模型

路径树模型适合更强调“按名称路径调用”的场景。

在这一层里，可以把块名看作一种路径。
例如 `A.B` 可以投影成对象路径 `A -> B`。

路径树模型的目标不是机械保留每一个中间分支，而是：

1. 在文档干净、无重复、无歧义时，优先给出自然的对象访问体验
2. 只有在真正发生重复或归属歧义时，才退化为数组

#### 默认行为

路径树模型默认按下面的顺序处理：

1. 如果某个路径可以唯一归属到一个已有父节点，则直接合并进去
2. 如果所需父节点不存在，则创建隐式父节点
3. 如果同一父节点下同名键重复，则该键变数组
4. 如果某个路径的父节点不再唯一，则该路径不再强行合并，而是按原始块顺序形成新的分支

这意味着：

1. `[A][A.B]` 应该保留为自然的 `A.B`
2. `[A.B][A.C]` 在无歧义时也应合并成同一个 `A`
3. `[A][A]` 或 `[A.B][A.B]` 这种重复路径会触发数组化
4. `[A][A][A.B]` 这种父节点不唯一的场景，不再强行挂到某个已有 `A` 上

#### 唯一父节点时的自然合并

例如：

```text
[A.B]
hello
```

可以被解释为：

```json
{
  "A": {
    "B": {
      "type": "text",
      "content": "hello",
      "order": 0
    }
  }
}
```

如果某个父节点本身也有正文，那么它也可以与子节点并列存在。

例如：

```text
[A]
root

[A.B]
child
```

可以被解释为：

```json
{
  "A": {
    "type": "text",
    "content": "root",
    "order": 0,
    "B": {
      "type": "text",
      "content": "child",
      "order": 1
    }
  }
}
```

这时用户仍然可以自然地通过 `A.B` 访问到子节点。

同样地：

```text
[A.B]
x

[A.C]
y
```

在无歧义时，也应被解释为：

```json
{
  "A": {
    "B": {
      "type": "text",
      "content": "x",
      "order": 0
    },
    "C": {
      "type": "text",
      "content": "y",
      "order": 1
    }
  }
}
```

#### 何时变数组

路径树模型不是“默认数组化”，而是“必要时才数组化”。

最直接的触发条件是：同一父节点下同名键重复。

例如：

```text
[A]
x

[A]
y
```

可以被解释为：

```json
{
  "A": [
    {
      "type": "text",
      "content": "x",
      "order": 0
    },
    {
      "type": "text",
      "content": "y",
      "order": 1
    }
  ]
}
```

再例如：

```text
[A.B]
x

[A.B]
y
```

可以被解释为：

```json
{
  "A": {
    "B": [
      {
        "type": "text",
        "content": "x",
        "order": 0
      },
      {
        "type": "text",
        "content": "y",
        "order": 1
      }
    ]
  }
}
```

也就是说：

1. 单次出现时，优先保持对象
2. 第二次及后续出现时，再升级为数组
3. 数组内顺序就是原始块顺序

#### 父节点不唯一时的分支退化

如果某个路径的父节点已经不唯一，那么继续强行合并会引入歧义。
这时路径树模型应当按原始块顺序形成新的分支，而不是猜测它属于哪个已有节点。

例如：

```text
[A]

[A]

[A.B]

[A.C]
```

可以被解释为：

```json
{
  "A": [
    {
      "type": "text",
      "content": "",
      "order": 0
    },
    {
      "type": "text",
      "content": "",
      "order": 1
    },
    {
      "B": {
        "type": "text",
        "content": "",
        "order": 2
      }
    },
    {
      "C": {
        "type": "text",
        "content": "",
        "order": 3
      }
    }
  ]
}
```

这里的 `B` 与 `C` 不应再被合并到前面的某个 `A` 对象中，而应保留为按顺序出现的独立分支。

因此，在这种解释方式下：

1. `A.B` 只有在无重复路径时，才天然接近普通对象访问
2. 一旦某层键重复，该层就会退化成数组访问
3. 一旦父节点不唯一，后续路径也可能退化成数组分支
4. 数组内对象不要求字段齐平，不需要为了结构整齐而补 `null`

### 16.3 元字段前缀

路径树模型需要保留一些元信息，例如：

1. `type`
2. `tag`
3. `content`
4. `order`

默认情况下，这些字段可以直接使用无前缀形式，以便前文示例更直观。

如果调用方担心与用户字段冲突，也可以在实现时给这些元字段统一加前缀。
推荐前缀是 `$`，例如：

```json
{
  "A": {
    "B": {
      "$type": "text",
      "$content": "hello",
      "$order": 0
    }
  }
}
```

如果调用方指定了前缀，那么在从路径树模型转换回其他模型时，也应使用相同前缀识别这些元字段。

如果调用方不使用前缀，而元字段名又与用户字段冲突，则应直接报错。

### 16.4 关于数字段

路径树模型不对数字段做特殊约定。

例如：

```text
[A.1]
x

[A.2]
y
```

可以被解释为：

```json
{
  "A": [
    {
      "1": {
        "type": "text",
        "content": "x",
        "order": 0
      }
    },
    {
      "2": {
        "type": "text",
        "content": "y",
        "order": 1
      }
    }
  ]
}
```

这里的 `1`、`2` 仍然只是普通路径段。
PML 在这一层不自动把它们解释成数组索引。

### 16.5 两种模型的关系

更准确地说：

1. 块序模型更接近核心解析结果，适合保真、存储、转换和调试
2. 路径树模型更接近面向调用的视图，适合在干净文档中按路径读取和组织数据
3. 路径树模型优先优化“无重复、无歧义”的自然对象体验
4. 当重复或歧义出现时，路径树模型会通过数组化保留更多信息，因此它通常是近似无损的，但不保证永远保持简洁对象访问

因此，PML 核心规范只负责提供稳定的块边界、名称系统与正文保留；
至于这些名称是否进一步投影成块序模型、路径树模型或其他数据结构，属于 PML 之上的可选解释层，而不是其核心定义。
